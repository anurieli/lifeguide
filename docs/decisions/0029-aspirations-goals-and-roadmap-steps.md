# 0029. Aspirations, Goals, and roadmap steps

**Status:** accepted (live) · **Date:** 2026-07-18 (landed 2026-07-20)

> **Landing note (2026-07-20).** This was written as "ADR 0022" on a branch that diverged from main before the 2026-07-18 backlog sweep; by the time it landed, 0022 through 0028 were taken, so it is renumbered 0029 with the content unchanged except as noted here. Two deltas from the original text: (1) `pillars-and-goals.md` was NOT resurrected as a live doc — main had already superseded it in favor of `pillars.md` (pillars are folders per ADR 0007/0022, not the tag model) and `goals.md`, which is now the single canonical goals doc; (2) the Coach's always-on goals context fragment rides alongside the pillars fragment main added in the meantime, not instead of it; (3) the "no production deployment carrying real goals data" premise was FALSE by landing day — prod held Todoist-synced goals in the old Orbit shape, and the schema push was rejected. Instead of clearing real data, `area`/`kind` are temporarily back as validate-only optionals and a one-shot `goals.migrateDropAreaKind` internal mutation strips them; once it has run against prod, the two schema lines can be deleted.

## Context

Ariel's read on the Goals page: it looked and behaved like a sync tool (Todoist projects with a task list), when the actual concept is much simpler and more personal — the few big things a person is chasing in life (a TED talk, climbing Everest), each of which deserves an honest look at what it'd actually take and a first move. Two things needed deciding before any code: what "aspiration" vs "Goal" even means as data, and whether this build should also resolve a bigger, already-parked question.

The live `goals` table (the "Orbit board," seeded from `_source-apps/goal-manager/Orbit-PRD.md`) had drifted from `docs/product/features/pillars-and-goals.md`'s own proposed shape for goals — the doc described `horizon`/`malleability`/`parentGoalId`/a three-state `status`, none of which were ever built; the real table shipped `area: business|personal|people`, `kind: big|shelf`, and a `status: active|planning|ongoing`, with no link to the `pillars` table at all. Separately, a live **Horizons** feature (`convex/horizons.ts`, a 5yr→1yr→1mo→week→today ladder) and a parked research note (`docs/research/raw/internal-notes/horizons-vision-goals-blend.md`, Linear ARI-103, "raw," 2026-07-15) already proposed unifying the vision board, Horizons, and Orbit goals into one measurable, deadline-bearing, board-anchored `goal` object — explicitly flagged there as "a big architectural change" to "pressure-test, not commit."

Ariel confirmed directly: do not resolve ARI-103 in this pass. It defers real calls (the vision-board anchor, a full table merge) that are his to make, not a default to infer mid-build.

## Decision

**The aspiration/Goal tier is purely `deadline` presence — one table, no separate object.** A `goals` row with no `deadline` is an aspiration (someday, dimmer in the gallery); setting a deadline graduates it into a dated Goal, clearing it ungraduates back. No redundant boolean or second table was introduced, so the two states can't drift apart from each other. `area`/`kind` were dropped outright (confirmed dead: no UI branch read either, only a default and one hardcoded Todoist-sync value wrote them) and replaced by an optional real link, `pillarId: v.id("pillars")` — the first actual foreign key between `goals` and `pillars`, where before the two tables shared only a conceptual doc, per `pillars-and-goals.md`'s own admission ("Pillars & Goals" is one conceptual element but two disconnected tables in code).

**A goal's roadmap is a small dependency graph in a new `roadmapSteps` table, not a flat checklist and not reused from `goalTasks`.** `goalTasks` stays exactly what it was (day-to-day, Todoist-synced Today/Inbox/Waiting execution items) — conflating it with a goal's own AI-scoped roadmap would have forced every roadmap step into Todoist-sync semantics it doesn't need. `roadmapSteps` carries `blockedBy: Id<roadmapSteps>[]`; whether a step is **blocked is computed at read time, never stored** — storing it would require fanning out a write to every dependent step on every status change, and would drift the moment that write was missed. Cycle handling is split by trust level: an AI-drafted batch silently breaks any cycle the model draws (same technique as `lib/thoughtMap.ts`'s existing cycle guard — first edge in input order wins, the closing edge is dropped), because untrusted model output degrading gracefully is the established posture (`parseDistilled`'s defaults-on-garbage is the same idea). A person's own live edit (`roadmapSteps.setBlockedBy`) is **rejected outright with a thrown error** instead — silently dropping a deliberate edit to real task data is a worse failure than a clear error telling them why it didn't stick.

**A goal/aspiration gets an async AI-drafted roadmap the moment it's created**, via a new `goalEnrich` node (`convex/ai/goalEnrich.ts`) in the cheap structuring tier (`openai/gpt-4o-mini`, same tier as `sessionDigest`/`thoughtMap` per ADR 0014 — a bounded, one-shot structuring task from a handful of fields, not the conversational tier). It copies `sessionDigest.ts`'s shape exactly: scheduled via `ctx.scheduler.runAfter(0, ...)` from `createGoal` so it never blocks capture, re-reads the goal fresh, and on any failure writes back an explicit `roadmapDraft.status: "error"` rather than leaving the card silently stuck.

**The Coach gets a real write path to Goals — a thin wrapper, not a new framework.** `coach.ask` was previously 100% advisory (its own system prompt said "you cannot place it yourself yet"); `docs/product/features/goals.md`'s claim that the Coach already saw goals context was traced and found stale/never built. A new always-included `goals.coachContext` fragment gives it real goal/pillar ids inline; a new cheap-tier `coachGoalIntent` classification runs before every reply and, if it recognizes a create/update intent, calls the **existing** `goals.createGoal`/`updateGoal` mutations directly — never a bespoke write path. It cross-checks any id the model returns against ids actually fetched that same call and defaults to a no-op on any mismatch, so it can never invent a goal or pillar to write to. This costs every Coach turn 2 model calls instead of 1 — an explicit, accepted tradeoff for staying a thin, always-on design rather than adding a keyword pre-filter's brittleness.

**ARI-103 was deliberately not resolved.** A goal gained one optional field, `laddersTo: five_year|one_year|one_month`, a one-way pointer at a Horizons standing rung — nothing more. The `horizons` table is untouched; no vision-board anchor field was added; the two tables were not merged. The fuller unification stays parked for Ariel to design when he's ready.

**Breaking schema change, dev-only, no migration mutation.** This repo has no production deployment carrying real goals data. Rather than writing a one-off internal mutation to strip `area`/`kind` from every existing row for the sake of preserving throwaway dev-test goals, the call is: clear the local `goals` and `goalTasks` tables before running `npx convex dev` against this schema (both tables together — clearing only `goals` would leave `goalTasks` rows pointing at deleted goals, silently orphaned in the Inbox view).

## Consequences

- **One canonical `goals` shape, matching what's actually built.** `docs/architecture/data-model.md` and `docs/product/features/pillars-and-goals.md` both described a `goals` table that was never real; both are rewritten in the same change to describe the actual shipped shape, closing a docs-drift gap rather than adding to it.
- **`pillars` and `goals` are now genuinely linked**, not just conceptually paired — a real step toward the "Pillars & Goals" element the docs already treat as one thing.
- **The roadmap can never silently drift into an unfinishable state from a person's own edit** (rejected cycles), while an AI-drafted batch degrades gracefully instead of erroring the whole draft over one bad edge.
- **Coach cost is real and explicit**: 2 model calls per turn, forever, unless a future pass adds a cheaper pre-filter. Flagged, not hidden.
- **ARI-103 remains open.** Anyone picking it up later inherits a real `laddersTo` hook to build from, but the vision-board anchor and full table merge are exactly as undecided as they were before this build.
- **Existing local dev goals data must be cleared once** before this schema lands in any dev deployment that already has rows — a one-time, explicit, documented step, not a silent break.
