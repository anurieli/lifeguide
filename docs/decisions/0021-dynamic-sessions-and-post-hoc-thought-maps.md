# 0021. Dynamic sessions and post-hoc thought maps

**Status:** accepted (live) · **Date:** 2026-07-18

## Context

ARI-18 asks whether the micro-interview is the single, universal way a person writes into their own model — and names one concrete slice: a session ("Thoughts") should be able to hold both a quiet brain dump and a guided, engaging conversation, in the same place. Today a session is exactly a container over immutable `captures` rows (ADR 0008) — raw truth on the captures, only derived display state (title/summary) on the container. The research note that seeded this work (`docs/research/wiki/raw/internal-notes/brain-dump-valve.md`) also asked whether an idea-graph pass over a dump belongs in the app; that shape was tried once already and tossed (ADR 0016, the Brain Dump Lab) because it was unreachable and mixed a live idea graph into the capture path itself.

Two shapes needed deciding: where does the interviewer's side of a dynamic conversation live relative to the capture spine, and where does a structured view of "what did I actually think" live relative to speech itself.

## Decision

**The conversation lives ON the capture spine, as a separate thin layer, not inside it.** Captures stay the immutable user raw exactly as ADR 0008 established; a new `sessionReplies` table holds the interviewer's side of the conversation, one row per turn, keyed to the session. This is the concept-graph-adjacent rebuild ADR 0016 prescribed ("if a concept-graph surface returns, it should be rebuilt against the current capture/session spine") — not a revival of the deleted lab, which mutated its own graph state inline with capture. A session gains `mode: "quiet" | "dynamic"` (absent = quiet) and an `interviewer` persona field (a hook, default `"coach"` applied in code — multiple personas are deliberately deferred). Mode only gates whether `sessionReply` (`convex/ai/sessionReply.ts`) generates a *new* turn; it never hides history, and it is switchable mid-session at any point. Generation is debounced 8s off each capture's ingest completion and self-superseding (a newer capture, or a reply that already covers the newest capture, makes the run no-op) — the same shape as the existing digest debounce, so a burst of quick appends still costs one model call.

**The thought map is post-hoc and regenerable, not live during speech.** `sessions.requestThoughtMap` is a deliberate, on-demand action ("Map my thinking"), not something that runs on every capture like the digest or the interviewer reply. One model call (`convex/ai/thoughtMap.ts`) reads the session's captures and returns a hierarchy of distinct thoughts; the result is stored as **one replaceable JSON row per session** in `thoughtMaps` — remapping patches the existing row in place, it never accumulates a history of maps. This is a v1 scope choice: a map that updated token-by-token as someone spoke would be a materially bigger feature (partial/unstable structure, much higher AI cost for something the person hasn't finished saying yet) for a payoff ("map my thinking so far") the post-hoc "Map my thinking" button already delivers on demand.

**The map reads user content only.** `assembleThoughtMapInput` and the query behind it (`sessions.getForThoughtMapInternal`) draw strictly from the session's own captures — never `sessionReplies`. The map is what *you* actually thought; the interviewer's questions are a scaffold for getting it out, not part of the thinking being mapped, and mixing them in would blur the whole point of "map MY thinking."

**Per-session maps now; cross-session merging deferred.** Each map is scoped to one session's captures. The node/edge shape was chosen so a later merge pass doesn't fight it: node ids are stable within a map, `level` and `rootId` are derived (not model-asserted structure baked permanently into storage), and a superseded thought is kept as a sibling rather than deleted — all properties a future "latching" pass (combining maps across sessions so a person's real recurring root theme emerges over time, per the original research note's framing) would need. Building that merge now would be speculative; nothing here blocks it later.

## Consequences

- **One archive, two thin layers.** `captures` remains the single immutable raw store sessions were built to protect (ADR 0008); `sessionReplies` and `thoughtMaps` are both derived/generated state that can be regenerated or dropped without touching a person's own words. Any future retroactive pass (decomposition, cross-session linking) still reads `captures` as the ground truth.
- **Quiet mode is unaffected byte-for-byte.** With `mode` absent (every session before this shipped, and any new session that never opts in), `sessionReply`'s first guard clause is a no-op — no new AI calls, no new rows, no behavior change. The feature is opt-in per session, not a redesign of the default experience.
- **No live idea-graph risk.** Because the map is a deliberate, on-demand, replaceable snapshot rather than a running structure updated inline with capture, this does not reintroduce the shape ADR 0016 tossed — the map cannot corrupt or race against the capture flow the way the old lab's inline graph could.
- **The interviewer persona and cross-session latching are both real gaps, tracked, not silently dropped.** `interviewer` is schema-ready for multiple personas but has no picker UI; the thought map's per-session scope is a known first cut, not a final answer to "what is my real root theme." Both are open questions in `docs/product/features/sessions.md`.
- **Cost stays bounded.** The interviewer reply reuses the Coach's own model tier (`anthropic/claude-sonnet-5`) because it runs on a tight per-turn loop and has to feel perceptive; the thought map reuses the digest's cheaper tier (`openai/gpt-4o-mini`) because it is a similarly-shaped structuring task run rarely, on demand, not on every capture.
