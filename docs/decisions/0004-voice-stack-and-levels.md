# 0004. Voice stack (provider-abstracted, OpenAI Realtime mini first) and Level model

**Status:** accepted (2026-06-03)

---

## Context

The onboarding rebuild (spec: `docs/superpowers/specs/2026-06-03-onboarding-rebuild-design.md`) introduced three decisions that needed to be settled before the code could be written:

1. **Which realtime voice model, and how tightly coupled should it be?** The Realtime API is OpenAI-specific; we wanted to avoid baking the model id and the provider into the UI.
2. **What exactly constitutes a "completed" blueprint, and what level does that earn?** The spec used the phrase "unlocked if not done" in a way that was internally contradictory; it needed a clean restatement.
3. **How does the QR phone handoff authenticate?** The phone is a different physical device; anonymous Convex identity is browser-cookie-bound, so the phone will not share the desktop's identity.
4. **Where should blueprint status be surfaced in the UI?** Options ranged from a persistent rail badge to a banner to nothing until the user asks.

---

## Decisions

### 1. Provider-abstracted voice with OpenAI Realtime "mini" as the first adapter

The voice stack lives in `convex/ai/voice/` and exports a `VoiceProvider` interface with a single `mint(instructions)` method. The first and only adapter is `OpenAIRealtimeAdapter`, which mints an ephemeral client secret at `POST /v1/realtime/client_secrets` (the GA Realtime endpoint; a nested `session` config carries the model + instructions) with `gpt-4o-mini-realtime-preview` (pinned in `convex/ai/config.ts` under the `"voice"` task key). Note: the original pre-GA `POST /v1/realtime/sessions` endpoint was retired by OpenAI and now 404s; the mint contract is pinned by `tests/voice-mint.test.ts`.

The model id is controlled by `TASKS.voice.model` in `config.ts`, the same mechanism used for all other AI tasks. Swapping to a different realtime provider requires:
- A new class implementing `VoiceProvider`.
- A one-line change to `getVoiceProvider()`.
- No UI changes.

The `OPENAI_API_KEY` Convex environment variable is required for voice. OpenRouter does not offer a Realtime API endpoint; the `openai` provider is used directly for this task.

### 2. All-18-boxes completion promotes to Level 1

A blueprint is "complete" when every one of the 18 `BLUEPRINT` questions has a non-empty, non-whitespace entry in `coreResponses`. "Filled by user" and "filled by synthesis" are equivalent; both count.

The level model:
- **Level 0:** blueprint unfinished (`blueprintStatus in {"unstarted", "in_progress"}`).
- **Level 1:** all 18 boxes filled (`blueprintStatus === "complete"`). This is the "app fully unlocked" milestone.
- **Level 2+:** deferred. The level-derivation function (`lib/levels.ts: deriveLevel`) is a stub that returns `1` for any complete blueprint. Ranking rules for higher levels will be defined when the engagement-driven progression system is designed.

`onboardedAt` is set independently at the end of onboarding regardless of blueprint completeness, so the user is never trapped in the wizard. Blueprint status is a separate, persistent signal that persists across sessions.

### 3. QR true join-token handoff

The QR URL encodes a short-lived, single-use join token: `${origin}/interview/${sessionId}?t=${token}`.

- The token is generated by `interview.issueJoinToken` (owner-only mutation): two concatenated `crypto.randomUUID()` calls.
- Only the SHA-256 hash of the token is stored in the `interviewSessions` row (`joinTokenHash`). The raw token is returned once, embedded in the QR, and never persisted.
- Public mutations (`markJoined`, `appendTurnByToken`, `endByToken`) accept a `(sessionId, token)` pair and verify by rehashing and comparing. They require no authentication.
- Tokens expire after 10 minutes (`joinTokenExpiresAt`).

This approach lets any phone join the session without signing into the account, at the cost of a 10-minute exposure window. For v1 this tradeoff is acceptable; cross-device auth hardening is a deferred item.

### 4. Status surfaced via calm Home banner and Guide marker (no rail badge)

Blueprint status is surfaced in two places, both unobtrusive:

- **Home banner** (`components/today/Today.tsx`): a single line of muted text above the morning/evening tabs: "Your blueprint isn't finished — N/18." with a "Continue" link. Hidden once the blueprint is complete. Not a toast, not a modal, not a persistent chip.
- **Guide marker** (`components/guide/Guide.tsx`): a small pill showing "Blueprint: N/18 | Level L". Informational, not a link.

No rail badge, no level chip in the header, no persistent notification. The design contract from [`../design/interaction-principles.md`](../design/interaction-principles.md) ("calm, never bombarding") is the governing rule.

---

## Consequences

- Every voice-capable onboarding session requires `OPENAI_API_KEY` in the Convex deployment. If the key is absent, `OpenAIRealtimeAdapter.mint()` throws; the `VoiceInterview` component catches this and offers the text interview fallback.
- Level 2+ progression is a stub. Any future ranking logic is a clean addition to `lib/levels.ts: deriveLevel` with no UI changes needed (the Guide marker already reads `settings.level` live).
- The join-token design means the phone route at `app/interview/[sessionId]` is entirely public (no auth wall). This is intentional for v1 friction reduction; a future hardening step could add a short PIN or require the same Google account.
- The all-18 threshold means synthesis alone cannot promote a user to Level 1 unless it fills every remaining box. Conflicts (where the AI could not fill a box because the user has authored text) keep boxes non-empty, so they count.
- Status surfacing is conservative (banner only). If user research shows people miss the incomplete-blueprint signal, escalating to a more prominent treatment is a one-component change.
