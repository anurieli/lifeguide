# 0024. The Core's Conversational mode: the mode machine, and mapping free speech to Blueprint keys

**Status:** accepted (live, 2026-07-18)

## Context

ARI-2 shipped in two slices. **Slice 0** (landed on main ~6 weeks ago) built the spine: `components/core/Core.tsx` grew a `grid | zen | conversational` mode switch, and `components/core/ConversationalCore.tsx` was a calm placeholder sharing the real Core data (`core.get`) and owning the exit/switch affordances, with the real engine explicitly deferred until `voice-field` landed on `main`. It has since landed: `components/voice/VoiceField.tsx`, `components/voice/SpeakSurface.tsx` (the Listener's always-on `/speak` call), `components/onboarding/VoiceInterview.tsx` (onboarding's realtime interview), `hooks/useRealtimeVoice.ts` (the shared WebRTC/transcription hook), and `convex/ai/voice/index.ts` (`mintRealtimeSession`). This ADR covers the remaining "heavy part": the real voice loop, mapping free-flowing speech onto the 18 fixed Blueprint keys, persistence, and the mode machine itself.

Two questions had to be answered:

1. **What is the mode machine, precisely?** Slice 0 already had three modes and ad hoc `useState` transitions in `Core.tsx`. Is grid required to mediate every switch, or can any mode reach any other directly (in particular Zen ↔ Conversational, called out explicitly in the original issue)?
2. **How does a free-flowing spoken conversation become answers to 18 specific, ordered questions?** Onboarding already solved a version of this for the *typed* interview (`lib/interview/policy.ts`'s `nextQuestion`, which asks the fixed policy's next key one at a time and tags every turn with that `questionKey`) and, separately, for the *voice* interview, which cannot tag turns to a key in real time because the conversation is open-ended — it instead runs `ai/synthesizeInterview.ts` once the call ends, mapping the whole transcript onto all 18 keys in one LLM pass.

## Decision

**1. The mode machine is a flat, fully-connected graph — no gating, no "must exit through grid."** Extracted into a pure reducer, `lib/core/mode.ts` (`coreModeReducer(state, action): CoreMode`), with three actions (`toGrid`, `toZen`, `toConversational`) that name the destination directly. `Core.tsx` now drives its mode via `useReducer(coreModeReducer, "grid")` instead of ad hoc `setMode` calls. This is the "mode-state" the issue asked to be tested — `tests/core-mode.test.ts` unit-tests the reducer directly (no React, no Convex): every mode can reach every other mode, in particular Zen → Conversational and back, without detouring through grid. The reducer's smallness is deliberate: since all three surfaces bind the *same* `coreResponses` (`api.core.get`/`api.core.save`), there is no mode-local state to reconcile on a transition, so the machine has nothing to gate.

**2. Conversational mode reuses onboarding's voice engine wholesale — not a fork.** `ConversationalCore.tsx` is built on exactly the same spine as `VoiceInterview.tsx` and `SpeakSurface.tsx`: `useRealtimeVoice` driving a WebRTC call over an `interviewSessions` row, minted by `ai/voice.mintRealtimeSession`. It reuses the **existing table** (`interviewSessions`, `experienceId: v.string()` — already untyped, so no schema change) with a new `experienceId` value, `"core"`, rather than a parallel table. On mount, it opens a fresh `interviewSessions` row (`interview.start({ experienceId: "core", device })`); every turn is appended via the existing `interview.appendTurn` (transcript-only — free-flowing turns carry no `questionKey`, so `appendTurn`'s existing "write straight to `coreResponses` when a turn IS tagged" fast path, used by the typed interview, simply never fires here). When the call ends, `interview.end` closes the session and `ai/synthesizeInterview.synthesizeInterview` — **the same action, unmodified** — reads the full transcript, drafts an answer per Blueprint key from what was actually said, and writes only the keys still empty (`applySynthesis`'s existing rule: an authored answer is never silently overwritten; a spoken answer that conflicts with one on file is returned as a `conflicts` key and surfaced in the summary screen, not applied). This is the "mapping free-flowing conversation output → specific blueprint keys" the issue called for — it existed already for onboarding's voice interview, and Conversational Core is a second caller of the identical mechanism, not a new one.

**3. The persona is personalized, since — unlike onboarding — this session can start from any fill state.** `mintRealtimeSession` grows one branch: for `experienceId === "core"` it calls a new pure function, `buildCoreInstructions(existingCore)` (exported from `convex/ai/voice/index.ts`, unit-tested in `tests/voice-core-instructions.test.ts` with no ctx/network), which reads the person's current `coreResponses` and builds a system prompt naming what's already filled (so the model never re-asks a settled question) and what's still open (so it favors that ground); once all 18 are filled it switches to a "reflect or revise, don't interrogate a checklist" framing. Every other `experienceId` is unaffected — `instructionsFor` still returns the fixed onboarding/Listener personas exactly as before.

**4. Switching modes mid-call ends the call first, but never blocks the switch.** `ConversationalCore`'s header (Zen / Grid) and its post-call "Continue in Zen" / "Done" both route through a `switchTo(target)` helper: if a call is live, it fires `voice.end()` (which runs `interview.end` then `synthesizeInterview`) and calls the mode-switch callback in the same tick, without awaiting. This is a judgment call: Convex mutations/actions aren't tied to a React component's lifecycle, so synthesis keeps running on the server even though the switch has already unmounted `ConversationalCore` — the grid/Zen views (both live `useQuery(api.core.get)` subscribers) pick up the newly-written answers reactively the moment synthesis lands, typically well under a second later. The alternative (block the switch on `await voice.end()`) would make "leave mid-sentence" feel like it hangs; this keeps the exit calm while keeping the data safe.

## Consequences

- **No schema change.** `interviewSessions`/`experienceEvents` already had untyped `experienceId: string`; `"core"` is a new value, not a new shape. `docs/architecture/data-model.md`'s comment enumerating known values is updated to say so.
- **The mapping mechanism has exactly one implementation for both callers.** Onboarding's voice interview and Core's Conversational mode both terminate in `synthesizeInterview`; a future improvement to the mapping (better prompting, partial-credit drafts, confidence scoring) benefits both automatically. The cost is that neither caller gets a Core-specific synthesis nuance without touching the shared action — acceptable today; if Core-specific needs diverge, the personalization point (`buildCoreInstructions`, already caller-specific per `experienceId`) is the seam to extend.
- **A person can talk over the same ground repeatedly.** Nothing marks a Blueprint key "settled by voice" differently from settled by typing or by the grid — malleability (`green`/`yellow`/`red`) already carries that weight, unchanged by this work.
- **Environment limitation (this change):** the realtime call itself (mic access, WebRTC, the actual OpenAI Realtime round-trip) could not be smoke-tested end-to-end in this sandbox (no live Convex deployment, no mic). The mint's request/response contract is pinned by the existing `tests/voice-mint.test.ts`; this change adds an action-level test (`tests/convex/core-conversational.test.ts`) that runs the real `mintRealtimeSession` handler (auth, session lookup, persona selection, telemetry) against convex-test's in-memory backend with `global.fetch` mocked, so everything up to the WebRTC handshake is verified without hitting the network. Manual voice smoke-test remains **UNVERIFIED**.
- **Incidental infra fix, needed to make the action test real.** convex-test's function-module discovery anchors an `import.meta.glob` to its own package location; in this multi-worktree sandbox `node_modules` is a symlink to a shared checkout, and Vite's default symlink handling resolved that glob against the *symlink target*, not this worktree — so `convexTest`-based action calls were silently exercising the shared checkout's code, not this branch's edits. Fixed by setting `resolve.preserveSymlinks: true` in `vitest.config.ts`, which is a no-op for a normal (non-symlinked) `node_modules` install, so it is safe outside this sandbox too. Worth other agent worktrees picking up if they hit the same silent-stale-test symptom.

## Tables touched

None new. `interviewSessions` and `experienceEvents` gain a new `experienceId` value (`"core"`); see `docs/architecture/data-model.md`.
