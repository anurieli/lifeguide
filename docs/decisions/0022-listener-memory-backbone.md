# 0022. The Listener's memory backbone: session-per-speaker, post-call summaries, tossed calls included

**Status:** accepted (live) · **Date:** 2026-07-18

## Context

ARI-23 named a real gap: the Listener (the `/speak` voice orb) has a face but no memory. The Center already answers *who is this person* (durable identity, filed into `coreFiles`); nothing answered *what have this person and I been talking about lately* — the conversational continuity that lets the orb open oriented instead of cold every single time. The research draft (`docs/research/listener-memory-backbone.md`, captured from a 2026-06-09 brainstorm) named this the **backbone**, distinct from the Center, and parked five load-bearing questions before any code. This ADR resolves all five and records what was built.

The single reshaping question was: **what is "a session per speaker," concretely?** The research doc offered three readings: (a) speaker-labeled turns within one call — LifeGuide already has this (`transcript[].role: "coach" | "user"`); (b) a separate rolling memory thread per human person; (c) multi-human diarization in a shared room.

## Decision

**1. "Session per speaker" = (b), realized trivially by the product's existing shape, not a new mechanism.** Every table in this app — `interviewSessions` included — is scoped to exactly one `userId`: one account, one human, gated by `getAuthUserId` everywhere. There is no concept anywhere in the schema of multiple human accounts sharing a call, and OpenAI Realtime gives one input audio stream per session — so (c), multi-human diarization, is out of scope; nothing in the product or the transport supports it, and building it would be speculative infrastructure for a capability nobody asked for. (a) is already built — `role` tags have existed since the onboarding interview shipped and needed nothing new. That leaves (b): the "session per speaker" IS the per-user chronological sequence of `interviewSessions` rows with `experienceId: "listen"`. Since a `userId` already picks out exactly one speaker, querying that user's own history (newest first) *is* their rolling memory thread — no new table, no new "speaker" concept, no per-thread bookkeeping beyond what the summary field below adds. `interview.latestListenSummaryInternal` (`convex/interview.ts`) is the concrete realization: it walks one user's own `interviewSessions`, newest first, and returns the most recent one with a done, non-empty summary.

**2. Summary cadence and writer: a dedicated post-call pass, separate from the Center.** `convex/ai/listenerMemory.ts`'s `summarizeSession` (the `listenerSummary` AI node) is its own isolated action, scheduled from `interview.end` (`ctx.scheduler.runAfter(0, ...)`) alongside — not inside — the Center's `synthesizeSession`. Splitting them was the point: the Center answers "what belongs in this person's file system," the backbone answers "what should the Listener remember about talking to them," and conflating the two would mean a toss (which must skip Core filing) also silently loses conversational memory, defeating half the point of "toss."

**3. Tossed calls still get a summary.** `interview.end`'s `status` argument was widened from `completed | abandoned` to also accept `tossed` (the schema already carried the `"tossed"` literal — `interview.ts`'s mutation just hadn't caught up to it). The summarize scheduling in `end` is unconditional on status: completed, abandoned, or tossed, any ended `"listen"` call gets summarized. Tossing discards the *Core* filing (a client-side choice about whether to run `center.synthesizeSession` at all) — it was never meant to mean "erase that we spoke." A toss button is not part of this change (schema comments indicate that UI is separate, in-flight work); the backend guarantee holds regardless of which client eventually calls `end` with which status.

**4. Memory depth: last session only, for v1.** The handoff (`convex/ai/voice/index.ts`'s `instructionsFor`) reads exactly one prior summary — the most recent done one — not the last N, and not a rolling synthesized "where we are" memo. A rolling memo is a materially bigger feature (it has to decide what survives across many calls, what decays, how contradictions between calls are handled) for a payoff ("pick up roughly where we left off") that a single well-written last-call summary already delivers for a first cut. This is the same "ship the smaller, correct thing first" call ADR 0021 made for thought maps (per-session now, cross-session "latching" deferred). **Open, tracked, not silently dropped:** multi-session memory (last N, or a synthesized standing memo) is a real future direction, listed in `listener.md`'s open questions.

**5. Relationship to the Coach's `threads`/`messages` and the Mirror: Listener-specific, least coupling.** The summary lives directly on `interviewSessions` (a field, not a new table, not a write into `threads`/`messages` or the Mirror). The research doc raised this as an open question without arguing either way; defaulting to no coupling keeps the blast radius of this change to exactly the Listener's own table, and keeps `threads`/`messages` free for whatever shape the text Coach eventually needs (still reserved, unbuilt). If the Coach and the Listener ever need to share continuity, that is a deliberate future merge, not an accidental one made by wiring them together now.

## Implementation shape

- **Schema:** `interviewSessions.summary?: { status: pending|done|error, text?, topics?, openThreads?, error?, at? }` — mirrors the `digest`-on-`sessions` pattern (ADR-adjacent to ARI-18's `sessions.digest`) rather than inventing a new shape. Only ever set for `experienceId: "listen"`.
- **The pass:** `convex/ai/listenerMemory.ts`'s `summarizeSession` (internalAction) reads the transcript via `lib/listenerMemory.ts`'s pure `assembleSummaryInput`, calls the `listenerSummary` AI node (`openai/gpt-4o-mini`, strict JSON — same cheap structuring tier as `sessionDigest`/`thoughtMap`), parses the result with pure `parseSessionSummary`, and writes it via `interview.writeSummaryInternal`. An empty transcript (opened and closed with nothing said) short-circuits to `status: "done"` with no text and no model call — a real, common, cheap-to-handle outcome.
- **The handoff:** `convex/ai/voice/index.ts`'s `instructionsFor` looks up `interview.latestListenSummaryInternal` for `"listen"` sessions only, and pure `lib/listenerMemory.ts`'s `buildListenerOpeningAddendum` turns that (or nothing) into text appended to `LISTENER_INSTRUCTIONS` via `agents/listener/persona.ts`'s new `buildListenerInstructions`. Onboarding sessions are untouched — `instructionsFor` returns the unmodified interviewer instructions for any non-`"listen"` `experienceId`.

## Consequences

- **No new table, no new "speaker" concept.** Resolving question 1 cost nothing structurally — the per-user scoping already in place on every table IS the per-speaker thread. This is a "the schema already does this" answer, not a build.
- **Toss and file are cleanly split.** A future toss affordance can call `interview.end({ status: "tossed" })` today and get full backbone continuity with zero Core writes — the split this ADR draws (decision 2/3) makes that free.
- **Cost stays bounded.** One cheap model call per ended Listener call, at most — same shape and tier as the digest/thought-map passes, not a new cost center.
- **Multi-session memory and multi-human diarization are both explicit, tracked gaps**, not silent omissions: see `docs/product/features/listener.md` §9.
- **`interview.end`'s status union is now `completed | abandoned | tossed`** — a client that only ever passed `completed`/`abandoned` is unaffected; a future toss UI needs no schema or mutation change to use it.
