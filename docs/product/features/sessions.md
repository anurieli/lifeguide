# Sessions (the living entry) — surfaced as **Thoughts**

**Status:** built (v3, 2026-07-12; dynamic mode + post-hoc thought maps added 2026-07-18, ARI-18; UX rework — plain-words toggle, map as a page view, auto-map, fit-to-screen graph, coach-opens-the-conversation, mic turn-states, no-dead-air — 2026-07-18; teachable map — per-user steering memo + Sonnet mapping — 2026-07-18, ARI-18; a committed text/quote capture reopens for editing — 2026-07-20, ARI-123; itemized brain-dump box — captures render as separate expandable cards, voice recordings get a play button, editing folded into the card — 2026-07-20, ARI-122) · **Element of:** Sessions (feeds the Core) · **Owns:** `sessions` (container rows); members are `captures` rows via `captures.sessionId`; also owns `sessionReplies` and `thoughtMaps`

> A session is one living journal entry: a brain dump you keep adding to. Tap ➕ and let it out; the entry can be reopened tomorrow to speak more, type into, drop photos or links on. Every piece stays a raw, immutable capture; the entry is the document view over them. The person can name the entry; the AI keeps a living description current for the list and for any agent traversing entries. **This is the single capture surface of the app** (v3, ADR 0010; the old flat-stream tab merged into it). The tab is labeled **Thoughts** (booklet icon — a session is a booklet; renamed from the short-lived "Dumps", v3.1 same day) and the main action exists on desktop as well as the phone: the phone's ➕, and on desktop a **scribbler pen** whose hover says **"Think out loud"**.

## 1. Purpose

The observation contract says: let the thought out first, structure later. The Thought Stream catches single thoughts; a session catches a *sitting*, the twenty-minute walk where one thread unspools across voice, a typed line, and a photo of the whiteboard. The roadmap's MVP capture spine names this entity ("Session: raw capture + metadata"). Because members stay ordinary captures with their raw artifacts kept forever, every future pass (decomposition into atomic thoughts, cross-session linking, pattern mining) can run retroactively over the whole archive; the session is the unit those passes will read.

## 2. User-facing behavior

**Starting one.** The main action creates a fresh session and lands inside its empty document. On the **phone** (bottom bar's big center ➕) it is already recording (no separate recording screen) — and the mic arms **before** the server answers with the new entry: the take's target is a promise the provider binds when creation resolves, so the first words never wait on the round-trip. On **desktop** (the pen in the rail below the nav items, and a pen in the list header — both titled "Think out loud") a plain click opens the document ready to type, with the mic one click away — auto-arming the mic on a mere click would throw a permission prompt unprompted. Tapping ➕ always starts a NEW entry — with one exception: from **inside an entry that is still empty** (no captures, no take saving into it), the main action reuses that entry instead of spawning an indistinguishable twin (on the phone it starts recording there; on desktop it's a no-op — you're already in a fresh page). Appending to an old entry goes through the list. If the mic is denied, a small note says "Mic unavailable. Tap the page and type." and the document stays fully usable. A too-short take (~1s) saves nothing.

**Never lose a take (2026-07-17).** A recording is decoupled from the entry it was started for: a take is *never* discarded because its entry failed to create. This closes a real hole — on a **cold start** (app just opened, or resumed after the phone locked / screen dimmed / PWA sat in the background), the stored auth token can be momentarily stale while it refreshes, so the very first `sessions.create` throws "Not authenticated". The old code treated that rejection as "no home for the take" and **cancelled the mic mid-sentence**, silently losing a whole ramble — while a second tap a beat later worked, because auth had settled. Two layers fix it: (1) the cold-start mutations (`sessions.create`, `files.generateUploadUrl`, `captures.create`) are wrapped in `lib/withRetry.ts` — short backed-off retries (~4.5s total) that ride out the token refresh; a thrown auth error is otherwise not retried by Convex. (2) The provider never cancels a live take on create failure; at save it resolves the entry, and if there still isn't one it **mints a fresh entry** for the take. Anything that still can't save is kept as a `failedTake` (blob in memory, never re-recorded) and **auto-retried on a gentle cadence** until it lands, on top of the existing manual "Try again". The only remaining loss is killing the app mid-take (parked crash-recovery buffer).

**Quick record (press ➕, swipe up).** Pressing the main action and swiping up — on the phone's ➕, the desktop rail pen, or the list-header pen — starts recording **the instant the gesture commits**, finger still down, on any device (the swipe is an explicit ask to record, so the desktop permission-prompt rule doesn't apply). Unlike a tap, the swipe **does not open the note**: the person stays on whatever screen they're on, the top-center recording pill appears from the very first frame (timer · ■ save), and the entry is created silently in the background (2026-07-15 — the first cut navigated into the note immediately, which made the swipe indistinguishable from a tap; Ariel: "it just immediately opens the note"). Tapping the pill's body opens the note; tapping its ■ saves the take right there without leaving the screen. Inside an empty entry the gesture records right there — no new note. If entry creation fails (a cold-start stale-auth blip, or offline), the recording is **not** dropped: the mic keeps running and the take lands in a fresh entry the provider mints at save (see "Never lose a take" below). The gesture lives in `hooks/usePressSwipeUp.ts` (pointer events, ~24px upward travel — kept short because iOS can cancel the pointer stream near the home-indicator zone — with the post-swipe synthetic click suppressed).

**The entry is an itemized box (ARI-122, 2026-07-20).** The session's captures render inside one visually contained box (`components/sessions/CaptureItem.tsx`, rendered from `SessionDoc.tsx`) — not a flowing blob of running text. Each capture is its own card, collapsed by default to a one-line preview (the transcript or "Listening back…"/error message for a recording, the text itself, the URL, or "Photo" with a small thumbnail for an image) plus a timestamp; **tapping the card expands it in place** to the full content — full transcript/text, the full-size photo, the clickable link — and tapping again collapses it. A voice recording's card carries its **own play button** (a small gold circular control, independent of the browser's native audio widget) that replays the stored audio without needing to expand the card first; pressing it never toggles the card. A **text or quote** card's expanded content is itself **click-to-edit** (see "Reopening a committed note" below) — clicking it swaps in a textarea in place, without toggling the card's own expand/collapse. Interleaved among the capture cards, in chronological order, are the interviewer's replies (dynamic mode) in their existing quieter style (indented, thin gold border, persona tag) — nested in the same box, not a card themselves, since a reply isn't captured input. Below the box sits the trailing editor: there is no composer strip and no send button, tap anywhere on the page and type; the paragraph commits as a text capture on blur (Cmd/Ctrl+Enter also commits), pasted links (a typed paragraph that is a single bare URL commits as a link capture, exactly as the old stream composer treated it) land as their own itemized card too. On desktop the mic (primary) and photo controls are a **horizontal pair that rides the text caret**: click anywhere and the two buttons sit just right of where you're writing — the record button is literally at the cursor, one click from the thought (`components/sessions/caret.ts` measures the caret via the mirror-div technique; the pair tracks focus, clicks, arrow keys, and typing, clamped to the page column). With no caret anywhere, the pair rests in flow below the box. While a take is live the pair swaps for the slim pill at the same spot. The document is pure capture on desktop: the Coach window and its floating buttons **step aside entirely** while an entry is open and return the moment you leave (2026-07-13, settled after two earlier layouts — corner-stacked, then side-by-side with the Coach — both by Ariel's live feedback). On the phone the controls keep floating bottom-right (no caret to ride). A live take shows in-flow inside the box as a pulsing timer row while the rest of the page stays usable, so you can type and drop photos mid-recording. While a take is live the mic swaps for one slim pill: discard (✕, two-tap "Sure?" confirm) · pause/resume (the timer freezes; paused stretches add no audio and no time) · save (■). **Saving is optimistic:** the moment you tap save, the take appears in the box as a pending row (gold dot · duration · "Processing…", the same shape as the live row it replaces) while the upload and capture write run behind it; the real capture card takes its place when the server has it, itself collapsed with "Listening back…" as its preview until transcription lands. The mic is free again immediately — the next take never waits on the previous one's upload. A small "What were you doing?" field holds context. Entries are never closed; reopening one days later and appending is the intended use.

**Reopening a committed note (2026-07-20, ARI-123; folded into the itemized box the same day, ARI-122).** A typed passage isn't locked once it's committed: expanding a text/quote capture's card and clicking its text (inside `CaptureItem.tsx`) reopens it in place as an auto-growing textarea, prefilled with its current text and the caret at the end — the same click-to-edit idiom as the Horizons/Rituals fields elsewhere in the app. Blur (or Cmd/Ctrl+Enter) saves the correction via `captures.update`; an unchanged or emptied-out edit is a silent no-op, so a stray click-then-blur never wipes the note. This is scoped to `rawType: "text" | "quote"` only — a voice take's transcript, an image's description, and a link's fetched text are all *derived* from the raw artifact, not something the person typed, so they stay read-only (retry-and-replace via "Try again" is still how those get corrected). Editing never fights the card's own expand/collapse or, for audio, its play button — clicks inside the edit field and the play control both stop propagation. Saving re-runs distillation over the new wording (the old `distilled`/`boardWorthy`/`embedding` were computed over the stale text) and, for a session member, bumps `sessions.updatedAt` and schedules the digest refresh on the same 30s debounce a fresh append uses.

> **Deferred (ARI-122 first slice):** the box currently expands/collapses each card independently (no "expand all"/"collapse all"); a card's expand/collapse state is client-only (not persisted, so it resets on reopen); and the play button has no seek/scrub bar or playback-rate control — it is play/pause only, same as before. None of these change the data shape; they are pure follow-on UI polish if requested.

**Name and living description (v3).** The header's title is an editable field: type a name and it is yours — `titleEditedAt` marks it person-entered and the AI never overwrites it (clearing the field hands naming back to the AI). Left blank, the digest titles the entry for the list. Beneath the header sits the **living description** (`summary`): re-synthesized as content lands (the debounced digest off each member's ingest) and additionally refreshed on **every open and leave** of the document (`sessions.refreshDigest`, which skips when the digest already covers the latest content, so repeated visits cost nothing). The description is the canonical line an agent pulls when traversing entries.

**Brain dump or Conversation (2026-07-18, ARI-18; reworded + reworked same day).** A slim strip under the title row carries the mode toggle and the Thoughts/Thought Map view switcher. The toggle reads **"Brain dump"** (mic icon, muted) by default — the entry is just held, no AI voice in the loop — or **"Conversation"** (speech-bubble icon, gold fill and border) once tapped; tapping again toggles back. This is plain-words UI over the same mechanism as before: `sessions.setMode` still writes `session.mode` directly with the stored values `"quiet" | "dynamic"` (absent counts as quiet everywhere it's read) — only the pill's label changed, not the data. **The coach opens the conversation.** The moment the toggle flips on, `sessions.setMode` schedules `ai/sessionReply.opener` (runAfter(0)): one short turn inviting the person to talk, hooking into whatever's already in the session if there's prior material, so the toggle visibly does something instead of just relabeling. It shares `insertReplyInternal`/`finishReplyInternal` and the same `sessionReply` model task as regular turns, with a trailing instruction telling the model this is an opener rather than a reply to new material. **No double-greeting:** it skips (no-ops) when the newest thread item (captures + replies merged by `createdAt`) is already a pending-or-done reply — flipping the toggle off and back on quickly doesn't stack openers; an *error* reply doesn't count as "already answered," so a session whose last attempt failed still gets a fresh opener next time. **In conversation mode**, after each appended capture finishes ingest, the interviewer takes one short turn — 2 to 4 sentences, one incisive question or a gentle pushback on vagueness, never a list, never a summary of the whole conversation (`convex/ai/sessionReply.ts`, the `sessionReply` node, `maybeReply`). **The wait varies by how the turn ended:** finishing a voice take is a deliberate end-of-turn, so it gets a fast ~2s turnaround; a typed capture might be mid-burst (more keystrokes coming), so it keeps the slower 8s debounce (`convex/ai/ingest.ts`). Both are debounced and self-superseding: a burst of quick appends collapses to one reply, because a scheduled reply checks whether a newer capture has landed (or whether a reply already covers the newest capture) and silently no-ops rather than firing a stale turn. The reply renders inline in the document, interleaved with the captures by timestamp: indented, a thin gold-left border, a small uppercase persona tag (`interviewer`, defaulting to **"coach"** — a hook for future personas like a business or physical coach; multi-persona selection is deliberately deferred), quieter type than the person's own material. While generating it shows "thinking…"; if the call fails it shows "Didn't catch that one." and the conversation just continues (no retry affordance on a single failed turn — the next capture's own reply attempt is the recovery). **No dead air:** the instant a take is saved (stopped, before it's even a real capture yet) or a text capture commits, a client-side-only "listening…" line appears at the bottom of the thread in the interviewer's own visual slot — bridging the gap before the real pending reply row shows up. It's shown only in conversation mode, only while nothing newer has answered yet (the newest capture is newer than the newest reply), never while still actively speaking, and it steps aside the moment a real reply row exists (pending, done, or error) — nothing is left dangling on failure. **The record button reflects the mode.** In Brain dump the ambient mic + photo controls are unchanged from before this rework. In Conversation the mic becomes the primary control with a visible turn-cycle (`ConversationMicButton` in `SessionDoc.tsx`, driven off `RecordingProvider`'s real state): idle **"Speak"** (gold, labeled pill, mic icon) → **recording** (the same discard/pause/stop pill as before, now gold-accented, the stop button pulsing) → **transcribing** (a disabled gold spinner badge while the take saves/uploads) → back to idle once the real capture lands. On the phone this is the main floating control; on desktop the same states apply to the caret-riding mic. **Switching modes never touches history** — replies already in the document keep rendering in quiet mode too (the backend just stops generating new ones); the toggle only gates *whether new turns get made* (including the opener), and it can be flipped mid-session at any point. The **Thoughts list** marks a dynamic entry with a small gold speech-bubble glyph beside its title (`SessionsList.tsx`), quiet entries carry none.

**The Thought Map is a view of the page, not a popup (reworked 2026-07-18).** The header strip carries a two-way switcher — **"Thoughts"** (the dump/conversation feed above) and **"Thought Map"** — that swaps the whole document body in place, on both mobile and desktop; there is no overlay, backdrop, or modal anymore, and the composer/recording controls never float over the map view (`ThoughtMapView` in `components/sessions/ThoughtMapView.tsx` no longer owns a close button or a fixed/backdrop wrapper — it's an ordinary flex child of `SessionDoc`). **The map now builds itself.** `convex/ai/ingest.ts` schedules `ai/thoughtMap.generate` 30 seconds after each session-member capture's ingest completes — the same debounce shape as the digest — so a session's map is usually already there by the time someone taps the Thought Map tab; no "Map my thinking" tap is required. To keep a burst of captures from costing one model call each, `generate` takes an optional `captureId` and **supersedes itself** the same way `ai/sessionReply.ts`'s `maybeReply` does: when `captureId` is present, it skips if a newer capture has landed in the session since (last capture wins). The manual path — `sessions.requestThoughtMap`, now surfaced as **"Map now"** (no map yet) or **"Remap"** (one exists) — schedules with `runAfter(0)` and *no* `captureId`, bypassing the guard, so an explicit request always runs. `writeThoughtMapInternal` was widened from patch-only to an upsert (insert if no row exists yet) so the very first write — which can now be the auto path, with nobody having tapped anything — still lands somewhere for the view to read. **View states:** no map yet → a calm empty state, "Your map builds itself shortly after you finish dumping." with a **Map now** button; pending → "mapping your thinking…"; error → the message with **Try again**; done → the map with a small **Remap** affordance top-right. The pass itself (`convex/ai/thoughtMap.ts`, the `thoughtMap` node) still reads **the person's own captures only, chronological, never the interviewer's replies** — even in a dynamic session, the map is what *you* actually thought — and returns a hierarchy: each node one distinct thought (a short label, an optional one-sentence detail), `parentId` for "part of / under" a larger thought, and edges (`leads_to`, `part_of`, `relates`) for how thoughts connect. A retracted thought ("never mind", "actually no", "scratch that") is marked **superseded** rather than deleted, and its replacement lands as a **sibling** (same parent), not a child of the retracted node — the map keeps the false start visible instead of erasing it. The model also names one **root theme** — the thing everything else in the sitting was really circling — returned as `rootId`; `lib/thoughtMap.ts`'s `normalizeThoughtMap` turns the untrusted JSON into the exact table shape (dedupes ids, drops dangling edges, clamps text, derives each node's `level` — depth from its nearest root — and picks the root with the most descendants if the model named more than one). **Desktop renders a hand-rolled layered SVG graph that always fits the screen** (`lib/thoughtMapLayout.ts` computes positions — y from `level`, x packed left-to-right per level so nothing ever overlaps, parents nudged toward the mean x of their children so the tree reads centered — plus a new `fitToContainer`, a pure function measuring the graph's bounding box against the rendered container and returning the scale that fits both dimensions via the SVG's viewBox, capped so a small map never magnifies past its natural size; a genuinely huge map clamps at a readable minimum, `MIN_READABLE_SCALE`, and falls back to pan/scroll instead of shrinking text past legibility — the default case always fits with no scroll). **Flow-first rendering:** `leads_to` edges are the visual spine — solid, gold, an arrowhead, the heaviest stroke; `part_of` is a light structural (tree) line; `relates` is the thinnest, dashed — all within the existing muted/stone/gold palette. The root node has a gold fill and border plus bold text (visually weightier than the rest), superseded nodes are dimmed with struck-through text, hovering a node shows its detail as a tooltip. **The phone** (and anywhere under the `md` breakpoint) hides the graph and instead renders the same data as a **collapsible outline** — indented by level, a chevron toggles a node's children, the root reads bold, superseded reads muted and struck-through, and each node's outgoing `leads_to` edges print as a small "→ leads to: …" line beneath it. Both views read the one stored row; nothing about the shape differs by device. A session with no user text yet to map errors with "nothing to map"; a malformed model response errors with a message and the same **Try again** path. The map is **per-session only for now**; merging thought maps across sessions so a person's real recurring root theme emerges over time ("latching") is an intended future pass — the schema (a `rootId` that can shift, nodes that stay stable ids within one map) was shaped not to fight that later, but it isn't built.

**Teach the map how you think (2026-07-18, ARI-18).** Next to Remap sits a small **"Teach it"** affordance (a graduation-cap icon). Tapping it opens an inline, calm editor — not a modal — right there in the map view: a textarea pre-filled with the person's current steering memo, a short helper line ("Tell the engine how you want your thinking mapped — fewer bigger nodes, what counts as the root, what to ignore. It follows this every time."), a plain **Save**, and a combined **Save & remap** that saves the memo then immediately fires a fresh `requestThoughtMap` so the change is visible right away. The memo (`settings.thoughtMapMemo`, per-user) is standing guidance, not a one-off instruction: every future generation for that person — auto-map and manual alike — folds it into the model's system prompt (`convex/ai/thoughtMap.ts`'s `generate`, via `lib/thoughtMap.ts`'s `buildMapSystemPrompt`), as a clearly-fenced section that tells the model to follow it even where it overrides the defaults. An empty or whitespace-only memo appends nothing — the unchanged default mapping. The same field is also surfaced in **Settings → The Coach** (`components/settings/ThoughtMapMemoField.tsx`, shared by both surfaces so the memo is findable and editable outside a session too, wired to the same `settings.update` mutation). Trimmed and capped ~2000 chars server-side.

**Recording survives leaving the screen.** The live take belongs to the shell (`RecordingProvider`), not the document: navigate to Today, the Board, or the list mid-take and it keeps recording. Anywhere outside the live entry, a quiet pill floats top-center (gold dot · elapsed · "Recording"/"Paused" · ■); tapping its body returns to the entry, tapping ■ saves the take right there. The pill shows even during quick record's brief window before the new entry's id is back from the server. Starting a new take anywhere (➕, or the mic in another entry) saves the current one into its own session first; there is only ever one live take.

**The list.** The **Thoughts** tab is the archive: pinned entries first, then newest by activity; relative time, a pin glyph, voice/photo count glyphs, the name (person-entered or AI), and the one-sentence living description (until the digest runs, the entry's first words). Tap to open. Swipe left to pin/unpin. Swipe right to delete: the row parks open with a Delete button, a second tap confirms. "Select" enters multi-select; picking 2+ entries shows "Merge N into one". The header carries its own pen ("Think out loud") so a fresh entry never requires leaving the surface.

**Demo thoughts.** The empty state offers "Show me two example thoughts", and a quiet "Add demo thoughts" link sits under a populated list. It seeds two fully packed entries — voice takes (with transcript and a playable file), photos, typed passages — so the surface can be seen lived-in. The media is painted client-side (`components/sessions/demoMedia.ts`: a synthesized soft-tone WAV via `lib/demoWav.ts`, canvas-drawn sunrise/notebook photos) and uploaded like any capture file; `sessions.seedDemo` then inserts ordinary sessions/captures with extraction already done and the digest stamped, so no ingest runs and no model is called over demo text. Demo rows are real data: open them, append, or swipe them away. Their `sourceMeta` carries `demo: true`.

**Merging.** Merge folds the selected sessions into the earliest-started one: every element keeps its own timestamp, so the merged document reads in the chronological order each piece was added, interleaved across the original entries. A person-entered name survives the merge (the earliest such session's name wins); AI title/summary are cleared and the digest re-synthesizes over the combined content.

**Deleting.** Deleting removes the entry from the archive view only: the container row is deleted, member captures are soft-deleted (`isActive: false`) with their raw artifacts and `sessionId` intact, so nothing raw is ever lost and future retroactive passes still see what belonged together.

**Desktop.** Thoughts is a rail tab (booklet icon, `NotebookPen`) and the rail carries the pen main action (`PenLine`, hover "Think out loud"); the same document behavior applies, and pin/delete appear as hover actions on list rows (no swipe). Desktop remains the command center. The phone bar is five even slots, Today · Board · ➕ · Thoughts · account, with the ➕ the only dark element (active tabs get a light tint, never a solid block); Core, the Talk/Listener tab, and the Atmosphere music player (the audio engine itself, not just the orb) are desktop-only; the phone is capture-first. The separate **Thoughts** tab is gone on every device (ADR 0010): the flat stream surface merged into this one, and its component (`components/thoughts/ThoughtStream.tsx`) stays in the codebase unrouted, like the Brain Dump Lab before it.

| Quick record | Press ➕/pen + **swipe up** (any device) | Mic arms the instant the gesture commits (`usePressSwipeUp`); the person stays put — no navigation — with the top pill live from the first frame while `sessions.create` runs behind the take; inside a still-empty entry it records there instead of creating | `sessions` (create), then as voice append |
| Expand/collapse a transcript | Tap a long spoken passage (or its Show more/less) | A transcript past ~320 chars renders clamped to 4 lines; tapping toggles the full text (view-only state, nothing stored) | read-only |

## 4. Dynamics with other elements

- **Thought Stream.** Merged into this surface as of v3 (ADR 0010): the flat stream is no longer a nav tab anywhere. Its **pipeline is unchanged and still load-bearing** — every member capture goes through the same ingest (transcription, link extraction, image reading) and distillation, and loose captures (`sessionId` absent: board intake, `voice.brainDump`) still work and surface in the board Inbox. See [`thought-stream.md`](thought-stream.md) for the pipeline; the stream *view* is retired pending a possible return as a filter/inspector over all captures.
- **Vision Board.** Unchanged. Session members surface in the board Inbox like any unplaced capture; a photo added to an entry from the phone is also inspiration waiting to be placed.
- **Journal (proposed).** The Journal's morning/night beats will *open* sessions (the roadmap's three front doors); the Journal owns `prompts` and writes into this table when built. See [`journal.md`](journal.md).
- **The Center / Core.** Not yet wired; sessions are upstream signal, like the stream.
- **The Listener.** Separate for now (`interviewSessions`); unifying Listener calls into this archive is an open question (see the listener memory backbone research note).
- **The Coach / interviewer persona.** Dynamic mode's `interviewer` field is a persona hook, not a picker: every session that turns dynamic on gets the default "coach" voice today (`convex/ai/sessionReply.ts`), with the field designed so future personas (a business coach, a physical coach…) can be assigned per session without a schema change. No persona selection UI exists yet.
- **Cross-session thought maps.** Each map is scoped to one session; nothing merges maps across sessions yet. The intended future ("latching") is that a person's real recurring root theme emerges by combining maps over time — deferred, tracked as an open question below.

## 5. States and edge cases

- **No AI keys:** entries and captures land; digest no-ops with fallback title; transcription errors show with retry; nothing is lost.
- **Failed save of a take (network):** the finished blob is kept in memory (by the provider, so it survives leaving the document) and the entry shows "That take didn't save; it's still here" with Try again, so retry never re-records. Saves being fire-and-forget, several takes can fail independently; they accumulate (`failedTakes`) and one Try again retries them all — and they also **auto-retry on a gentle cadence** so a take lands even without the person tapping. Killing the app mid-take still loses it (accepted; crash-recovery buffer is a parked follow-up).
- **Cold start / entry creation fails (stale auth or offline):** the take is **never** dropped. On a cold or resumed start the first `sessions.create` can throw "Not authenticated" while the auth token refreshes; the create is retried (`lib/withRetry.ts`) and, whether or not that lands, the mic keeps recording. At save the provider resolves the entry or **mints a fresh one**, and anything still unsaved waits in `failedTakes` and auto-retries. This replaced the old behavior, which cancelled the mic on create failure and silently lost a whole first-of-session ramble (Ariel, 2026-07-17).
- **Take saving while its entry looks empty:** an entry whose only content is a take still uploading (or one that failed and awaits retry) is not a husk — the back action and the list sweep both leave it alone until the save resolves.
- **Mic dies mid-session (or never starts):** a synchronous `MediaRecorder.start()` failure is caught and surfaces as the mic-unavailable note instead of an unhandled error; the take target clears so no zombie "Recording" pill survives.
- **Failed transcription:** the audio file is stored regardless; the entry says "the recording is safe" with Try again.
- **Typed text pending on leave:** the back action commits a non-empty draft as a capture instead of husk-checking; a failed commit puts the text back on the page.
- **Empty session:** the ➕ flow creates the container before any content, so bailing out immediately is the common path; `deleteIfEmpty` clears it on the entry's back action, and the list sweeps any zero-count row on render.
- **Swipe vs tap vs scroll:** rows use `touch-action: pan-y` (vertical scroll stays native); a horizontal swipe suppresses the click that browsers fire after touchend, so a swipe never opens the entry.
- **Digest burst:** several appends in a row cost one model call (each ingest completion schedules a run 30s out; runs skip while any member is pending, and overwrite idempotently).
- **Reply burst (dynamic mode):** several quick appends in a row also cost one model call, by the same shape as the digest debounce — each capture's ingest schedules a reply (2s for a voice take, 8s otherwise) out, and a run no-ops the moment a newer capture has landed (that capture's own scheduled run supersedes it) or the newest reply already covers the newest capture.
- **Mode is quiet by default and per-session:** a brand-new entry, and every entry before 2026-07-18, has `mode` absent, which reads as quiet everywhere. Switching to dynamic and back never deletes or hides replies already generated; it only gates whether new ones (including a fresh opener) get made.
- **Toggling on and off quickly doesn't spam openers:** `opener`'s guard reads the newest thread item at run time, not at schedule time — if a reply already answered by the time it runs (or mode flipped back off), it no-ops.
- **A failed interviewer reply doesn't retry itself:** the pending row is marked `error` and renders "Didn't catch that one." with no retry button on that specific turn — the conversation simply continues, and the next appended capture gets its own fresh attempt at a reply. An opener is the one exception: a failed opener doesn't block a *later* opener attempt (toggling off and back on), since an error reply doesn't count as "already answered" in that guard.
- **Auto-map burst:** the same shape as the reply/digest debounce — every session-member capture's ingest schedules `ai/thoughtMap.generate` 30s out with its own `captureId`; a run no-ops the moment a newer capture has landed in the session (last capture wins), so a burst of appends costs one model call for the map too.
- **Thought map with nothing to map:** a session with no extractable user text yet (e.g. only a still-transcribing take) writes `thoughtMaps.status = "error"` with `"nothing to map"`; the Thought Map view shows the message with Try again. This applies to both the auto and manual paths.
- **No steering memo (default):** `settings.thoughtMapMemo` absent or whitespace-only means `buildMapSystemPrompt` returns the base prompt unchanged — mapping behaves exactly as it did before the teachable-map feature.
- **Malformed model output:** `normalizeThoughtMap` validates and clamps everything the model returns (dedupes node ids, drops edges pointing at missing/invalid nodes, breaks parent-chain cycles rather than recursing forever); if the response is unusable outright (no valid nodes array) the map writes `status: "error"` with a message, surfaced the same way with Try again.
- **A huge map still fits, or falls back honestly:** `fitToContainer` never magnifies a small map past its natural size (it just sits centered/letterboxed) and never shrinks a large one past `MIN_READABLE_SCALE`; past that point the graph is rendered at the clamped (still-readable) size and the container becomes pan/scrollable instead of squeezing text further.
- **iOS Safari:** `useAudioRecorder` negotiates audio/mp4; ingest maps the extension. On-phone QA tracked in `TO-CHECK.md`.

## 6. AI involvement

`sessionDigest` (`convex/ai/config.ts`): OpenRouter, JSON `{title, summary}`, grounded strictly in member text via `lib/sessionDigest.ts` (`assembleDigestInput`, 6k cap, chronological, labeled by kind). Members' transcription/vision/distillation are the existing Thought Stream nodes.

**`sessionReply`** (ARI-18, `convex/ai/sessionReply.ts`): OpenRouter, `anthropic/claude-sonnet-5`, temperature 0.7 — the same tier as the Coach's own live chat, since this runs on a tight per-turn loop and has to feel perceptive, not templated. Plain text out (no JSON): one turn, 2–4 sentences, grounded in `buildReplyMessages` — the capture/reply history chronologically merged into `user`/`assistant` turns (captures still mid-extraction are skipped; capped to the last 40 items). System prompt (in `convex/ai/config.ts`) casts the model as a sharp, warm interviewer, not a therapist or a coach reciting affirmations: mostly asks the one most incisive next question, pushes back gently on vagueness, mirrors the person's register, and gets out of the way (a bare "go on") when they're clearly mid-flow. **`opener`** (same file, UX rework 2026-07-18) is the same `sessionReply` task and system prompt, called with `buildReplyMessages` plus one trailing `user` instruction telling the model this is a conversation-opening turn, not a reply to new material — everything else (logging, error handling, the `sessionReplies` row it writes into) is identical to a regular turn.

**`thoughtMap`** (ARI-18, `convex/ai/thoughtMap.ts`): OpenRouter, `anthropic/claude-sonnet-5`, temperature 0.3, strict JSON mode — bumped from `openai/gpt-4o-mini` (2026-07-18, teachable map): mapping quality is judgment-bound (what's a distinct thought, where the real root sits, what got retracted), so it earns the same per-node tiering as the app's other judgment-bound nodes (ADR 0014) rather than sitting with the cheap structuring tasks. Runs automatically in the background (UX rework 2026-07-18: `convex/ai/ingest.ts` schedules it 30s after each session-member capture's ingest, same debounce shape as the digest) as well as on demand (`sessions.requestThoughtMap`, the "Map now"/"Remap" affordance). Input is `assembleThoughtMapInput`: the session's own captures only, chronological, capped to 8k chars. The system prompt is no longer the task's static config string alone: `generate` loads the person's steering memo (`settings.thoughtMapMemo`, via `settings.getMemoInternal`) and folds it in with `lib/thoughtMap.ts`'s `buildMapSystemPrompt` — a pure function that appends the memo as a clearly-fenced, explicitly user-authored section when non-empty, and returns the base prompt unchanged otherwise (unit-tested in `tests/thought-map-memo.test.ts`). Output is `{nodes, edges, rootId}` per the system prompt's contract (node `id`/`label`/`detail?`/`parentId?`/`status`; edge `from`/`to`/`kind`/`label?`); `lib/thoughtMap.ts`'s `normalizeThoughtMap` is the pure validation/normalization layer between the untrusted model JSON and the `thoughtMaps` table (unit-tested independently of the model in `tests/thought-map-normalize.test.ts`).

## 7. Data touched

`sessions { userId, title?, titleEditedAt?, summary?, doing?, device: phone|desktop, digest?{status: pending|done|error, at?}, mode?: quiet|dynamic, interviewer?, startedAt, updatedAt, pinnedAt?, lastOpenedAt? }` (index `by_user_updated`); `captures.sessionId?` (index `by_session [sessionId, createdAt]`). `titleEditedAt` marks the name person-entered; the digest write guards on it. `mode` gates whether `sessionReply` generates new turns (absent = quiet); `interviewer` is the persona hook, defaulting to "coach" in code when absent. Element-level metadata rides on each capture: `createdAt` (exact add time), and for audio `sourceMeta` JSON carries `device`, `durationMs`, `recordingStartedAt`.

`sessionReplies { userId, sessionId, afterCaptureId?, persona, status: pending|done|error, text?, error?, createdAt }` (index `by_session [sessionId, createdAt]`) — one row per interviewer turn; `afterCaptureId` is the capture that triggered it.

`thoughtMaps { userId, sessionId, status: pending|done|error, error?, nodes[{id, label, detail?, level, status: active|superseded, parentId?}], edges[{from, to, kind: leads_to|part_of|relates, label?}], rootId?, generatedAt, createdAt }` (index `by_session [sessionId, createdAt]`) — one live, replaceable row per session (patched in place by `sessions.requestThoughtMap`, never a second row inserted for the same session).

`settings.thoughtMapMemo?` (ARI-18 teachable map) — not owned by Sessions (it lives on the system-level `settings` table, see [`../../architecture/data-model.md`](../../architecture/data-model.md)), but it's the standing input the thought map reads on every generation; edited from both this element's "Teach it" panel and Settings.

See [`../../architecture/data-model.md`](../../architecture/data-model.md), [ADR 0008](../../decisions/0008-sessions-as-container-over-captures.md), and [ADR 0021](../../decisions/0021-dynamic-sessions-and-post-hoc-thought-maps.md).
