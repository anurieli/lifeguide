## 2026-07-15 · Feedback: one-click "screenshot this page" (a third attach option) + smoother photo uploads

Ariel's report from a screenshot of the Feedback composer: attaching a photo felt broken ("doesn't show it's there or in the payload"), and he wanted to add a screenshot of the current page in a single click — a third option beside mic and attach. ## 2026-07-14 · Todoist sync actually works again: migrated off the sunset REST v2 API to unified v1

Ariel's screenshot: the Goals board's "Sync Todoist" button threw `[CONVEX A(todoist:sync)] … Server Error` even after the token was confirmed correct. Root cause found by probing the endpoints directly — **not** a token problem: Todoist has **sunset the REST v2 API**. Every call in `convex/todoist.ts` hit `https://api.todoist.com/rest/v2/*`, which now returns **HTTP 410 Gone** (verified live; the new unified `…/api/v1/*` returns 401, i.e. alive and just needs auth). No token can fix a deleted endpoint — and the previous "surface the real error" fix couldn't reveal it because Convex **redacts** thrown `Error` messages to a bare "Server Error" in production, which is exactly what the screenshot showed. Fix: migrated the whole file to the **unified v1 API** (`https://api.todoist.com/api/v1`). v1's two breaking differences are handled: (1) list endpoints (`/projects`, `/tasks`) now return a paginated `{ results, next_cursor }` envelope capped at 200/page instead of a bare array — a new `todoistList()` helper unwraps the envelope and follows `next_cursor` to the end (and still tolerates a bare array defensively); (2) field extraction now falls back across v1 naming (`order ?? child_order`, `is_inbox_project ?? inbox_project`). The connect-test (`saveToken`) and push actions (`pushChecked`, `pushCreate`) ride the same base + relative paths, so they move to v1 too. Verified: live endpoint probes (v2→410, v1→401), and a standalone logic test driving `todoistList` + the sync mapping through a mocked two-page v1 response — pagination/cursor-follow, envelope unwrap, field mapping, defensive fallbacks, and bare-array tolerance all pass (11/11 assertions). Changes: `convex/todoist.ts` only.

**Docs touched:** `docs/product/features/goals.md` (Pull bullet: v1 base URL, the v2→410 sunset, and the paginated envelope + `todoistList` cursor-follow).

## 2026-07-13 · Sign-in sticks on the phone: auth cookies persist for a rolling 14 days

Ariel's report: every reload of the app on his phone demanded a fresh Google sign-in. Root cause: Convex Auth's Next.js integration keeps the auth tokens in cookies set by `middleware.ts`, and with no `cookieConfig` those are **session cookies** — deleted whenever the browser session ends, which mobile browsers do aggressively every time the tab/process is killed (the library's own docs: "If this is not set, the cookie will be a session cookie"). Desktop browsers restore sessions, which is why the bug only bit on mobile. Fix: `middleware.ts` now passes `cookieConfig: { maxAge: 60 * 60 * 24 * 14 }` (14 days, Ariel's chosen window) to `convexAuthNextjsMiddleware`. The middleware re-issues the cookies on each authenticated request, so it's a rolling window — sign-in is only demanded again after 14 days without opening the app; server-side session durations stay at the library's 30-day defaults, so the cookie is the binding limit. Anonymous identities ride the same cookies and get more durable on mobile too. Changes: `middleware.ts` only. tsc clean.

**Docs touched:** `docs/architecture/security-privacy.md` (Auth → new "Session persistence: a rolling 14-day window" subsection).

## 2026-07-13 · Todoist token: save now tests the connection for real, and shows the error inline

## 2026-06-04 · Fix: Google login token not persisting on prod (`SITE_URL` mismatch)

- **Why it broke.** Convex Auth runs OAuth entirely on `<deployment>.convex.site`, then redirects the browser back to the app (where the Next.js middleware sets the `__convexAuthJWT` / refresh cookies) using `SITE_URL`, rejecting any other origin (`@convex-dev/auth/.../implementation/redirects.js`). The app calls `signIn("google")` with no `redirectTo`, so it always fell back to `SITE_URL`. With `SITE_URL=localhost:3000`, prod OAuth bounced the browser to localhost and the token never landed on `mylifesguide.com`. Dev worked because it matched. Vercel logs were a red herring — the failure is server-side on Convex, and the only Vercel project the MCP can see is an empty placeholder (real prod is the `lifeguide` project on the other Vercel account).
- **The fix (one shared deployment serves both dev and prod).** Set `SITE_URL=https://mylifesguide.com` on `gregarious-boar-475`; `convex/auth.ts` now supplies a `callbacks.redirect` whitelisting the approved app origins (`mylifesguide.com`, `www.mylifesguide.com`, `localhost:3000`); `components/auth/StartButton.tsx` passes `redirectTo: window.location.origin` so each host gets the token back. Prod login is fixed by the env change alone (old prod button falls back to the now-correct `SITE_URL`); the client + callback changes restore local-dev Google login and make prod robust across apex/www.
